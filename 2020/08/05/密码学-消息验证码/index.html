密码学-消息验证码

Basic concepts

Message integrity

Message integrity (Message authentication): each party should be able to identify when a message it receives was sent by the party claiming to send it, and it was not modified in transit.

E.g. A bank should make sure a transfer request is sent by a legitimate user, and the received request should not be modified.

Remember that encryption does not privide any integrity unless it is specifically designed with that purpose!

E.g. For a stream cipher, the attacker can filp one or more bits of the ciphertext ccc to modify the decrypted plaintext. Since the attacker do not know the plaintext, it does not contradict the concept of secrecy.

Message Authentication Codes (MAC)

Definition

A message authentication code (MAC) consists of three probabilistic polynomial-time algorithms (Gen,Mac,Vrfy)(Gen,Mac,Vrfy)(Gen,Mac,Vrfy) such that

  1. The key-generation algorithm GenGenGen takes as input the security parameter 1n1^n1n and outputs a key kkk with ∣k∣≥n|k|\ge n∣k∣≥n.
  2. The tag-generation algorithm MacMacMac takes as input a key kkk and a message m∈{0,1}∗m\in \{0,1\}^*m∈{0,1}∗, and outputs a tag ttt. We write it as t←Mack(m)t\leftarrow Mac_{k}(m)t←Mack​(m) since the algorithm might be randomized.
  3. The deterministic verification algorithm VrfyVrfyVrfy takes as input a key kkk, a message mmm and a tag ttt. It outputs a bit bbb, with b=1b=1b=1 meaning valid and b=0b=0b=0 meaning invalid. We write this as b:=Vrfyk(m,t)b:=Vrfy_{k}(m,t)b:=Vrfyk​(m,t).

It is required that for every nnn, every key kkk output by Gen(1n)Gen(1^{n})Gen(1n) and every m∈{0,1}∗m\in \{0,1\}^*m∈{0,1}∗, it holds that Vrfyk(m,Mack(m))=1Vrfy_{k}(m,Mac_{k}(m))=1Vrfyk​(m,Mack​(m))=1.

Security of MACs

The message authentication experiment Mac−forgeA,π(n)Mac-forge_{\mathcal{A},\pi}(n)Mac−forgeA,π​(n):

  1. A key kkk is generated by running Gen(1n)Gen(1^{n})Gen(1n).
  2. The adversary A\mathcal{A}A is given input 1n1^n1n and oracle access to Mack(⋅)Mac_{k}(\cdot)Mack​(⋅). The adversary eventually outputs (m,t)(m,t)(m,t) (mmm could be any messages of its choice). Let Q\mathcal{Q}Q denote the set of all queries that A\mathcal{A}A asked its oracle.
  3. A\mathcal{A}A succeeds if and only if (1) Vrfyk(m,t)=1Vrfy_{k}(m,t)=1Vrfyk​(m,t)=1 and (2) m∉Qm \notin \mathcal{Q}m∈/​Q. In that case the output of the experiment is defined to be 1.

A message authentication code Π=(Gen,Mac,Vrfy)\Pi =(Gen, Mac, Vrfy)Π=(Gen,Mac,Vrfy) is existentilly unforgeable under an adaptive chosen-message attack, or just secure, if for all probabilistic polynomial-time adversaries A\mathcal{A}A, there is a negligible function neglneglnegl such that

Pr[Mac-forge_{\cal{A},\pi}(n)=1]\le negl(n)

This definition of security is strong in two respects. First, the adversary is allowed to request MACMACMAC tags for any messages of its choice. Second, the adversary is considered to have broken the scheme if it can output a valid tag on any previously unauthenticated message.

By making the definition of security for MACs as strong as possible, we ensure that secure MACs can be used for a wide range of purposes without worring about compatibility of the MAC with the semantics of the application.

Remember the MAC itself offers no protection against replay attacks! To prevent replay attacks, two common techniques are used:

  • Sequence numbers (or counters): the communicating users maintain(synchronize) a state. It can be problematic when users communicate over a lossy channel.
  • Time stamps: the sender prepends the current time TTT to the message before authenticating, and sends TTT along with the mesage and the and the resulting tag ttt. The receiver verifies that ttt is a valid tag for T∣∣mT||mT∣∣m and TTT is within acceptable clock skew from current time. This method requires maintaining closely synchronized clocks. And a replay attack can still take place if it is done quickly enough.

Strong MACs

Consider a modified experiment Mac−sforgeMac-sforgeMac−sforge that defined the same way as Mac−forgeMac-forgeMac−forge, except that the set Q\mathcal{Q}Q contains pairs of oracle queries and their associated responses (i.e. (m,t)∈Q(m,t)\in \mathcal{Q}(m,t)∈Q). The adversary A\mathcal{A}A succeeds if and only if A\mathcal{A}A outputs (m,t)(m,t)(m,t) such that Vrfyk(m,t)=1Vrfy_{k}(m,t)=1Vrfyk​(m,t)=1 and (m,t)∉Q(m,t)\notin \mathcal{Q}(m,t)∈/​Q.

A message authentication code Π=(Gen,Mac,Vrfy)\Pi=(Gen,Mac,Vrfy)Π=(Gen,Mac,Vrfy) is strongly secure, or a strong MAC, if for all probabilistic polynomial-time adversaries A\mathcal{A}A, there is a negligible function neglneglnegl such that:

Pr[Mac-sforge_{\cal{A},\pi}(n)=1]\le negl(n)

Let Π=(Gen,Mac,Vrfy)\Pi=(Gen,Mac,Vrfy)Π=(Gen,Mac,Vrfy) be a secure MAC that uses canonical verification (the receiver re-compute the tag and check for equality). Then Π\PiΠ is a strong MAC. (Attention that the definition of the MAC do not have any restriciton to how VrfyVrfyVrfy algorithm works.)

A potential timing attack (an example of side channel attack): Consider an adversary who can send message/tag pairs to the receiver. The adversary can learn not only whether the receiver accepts or not, but also the time to make decisions. If there is a MAC using canonical verification, the time to reject differs depending on the position of the first unequal byte. The attacker can send (m,t0),⋯ ,(m,t255)(m,t_{0}),\cdots,(m,t_{255})(m,t0​),⋯,(m,t255​) to the receiver, where tjt_{j}tj​ is a string with the first iii bytes set correctly and the (i+1)st(i+1)st(i+1)st byte equal to jjj. The tag with slightly longer time for rejection has the correct (i+1)st(i+1)st(i+1)st byte.

MAC verification should use time-independent string comparison that always compares all bytes!

Construct secure MACs

A fixed-length MAC

Construction

Let FFF be a pseudorandom function. Define a fixed-length MAC for messages of length nnn as follows:

  • Mac: on input a key k∈{0,1}nk\in \{0,1\}^nk∈{0,1}n and a message m∈{0,1}nm\in \{0,1\}^nm∈{0,1}n, output the tag t:=Fk(m)t:=F_{k}(m)t:=Fk​(m). (If ∣m∣≠∣k∣|m|\ne |k|∣m∣​=∣k∣ then output nothing.)
  • Vrfy: on input a key k∈{0,1}nk\in \{0,1\}^nk∈{0,1}n, a message m∈{0,1}nm\in \{0,1\}^nm∈{0,1}n and a tag t∈{0,1}nt\in \{0,1\}^nt∈{0,1}n, output 1 if and only if t=Fk(m)t=F_{k}(m)t=Fk​(m). (If ∣m∣≠∣k∣|m|\ne |k|∣m∣​=∣k∣ then output 0.)

If F is a pseudorandom function, then the above construction is a secure fixed-length MAC for messages of length n.

Proof

Consider the MAC Π~\tilde{\Pi}Π~ using a truly random function fff instead of pseudorandom function FkF_{k}Fk​. For any message m∉Qm\notin \mathcal{Q}m∈/​Q, the value t=f(m)t=f(m)t=f(m) is uniformly distributed in {0,1}n\{0,1\}^n{0,1}n from the view of adversary A\mathcal{A}A, so we can get:

Pr[Mac-forge_{\cal{A},\tilde{\pi}}(n)=1]\le 2^{-n}

Then construct a distinguisher DDD that is given input 1n1^n1n and access to an oracle O:{0,1}n→{0,1}n\mathcal{O}:\{0,1\}^{n}\rightarrow \{0,1\}^{n}O:{0,1}n→{0,1}n. (1) Run A(1n)\mathcal{A}(1^{n})A(1n). Whenever A\mathcal{A}A queries its MAC oracle on a message mmm, A\mathcal{A}A obtain a response ttt. (2) When A\mathcal{A}A outputs (m,t)(m,t)(m,t) at the end of its execution, query O\mathcal{O}O with mmm and obtain response t^\hat{t}t^. If t^=t\hat{t}=tt^=t and A\mathcal{A}A never queried its MAC oracle on mmm, then output 1; otherwise output 0.

If DDD's oracle is a pseudorandom function, then the view of A\mathcal{A}A when run as a sub-routine by DDD is distributed identically to the view of A\mathcal{A}A in experiment Mac−forgeA,π(n)Mac-forge_{\mathcal{A},\pi}(n)Mac−forgeA,π​(n). Therefore,

Pr[D^{F_{k}(\cdot)}(1^{n})=1]=Pr[Mac-forge_{\cal{A},\pi}(n)=1]

If DDD's oracle is a truly random function, then the view of A\mathcal{A}A when run as a sub-routine by DDD is distributed identically to the view of A\mathcal{A}A in experiment Mac−forgeA,π~(n)Mac-forge_{\mathcal{A},\tilde{\pi}}(n)Mac−forgeA,π~​(n). Therefore,

Pr[D^{f(\cdot)}(1^{n})=1]=Pr[Mac-forge_{\cal{A},\tilde{\pi}}(n)=1]

Since FFF is a pseudorandom function and DDD runs in polynomial time, so there exists a negligible function neglneglnegl that

∣Pr[DFk(⋅)(1n)=1]−Pr[Df(⋅)(1n)=1]∣≤negl(n) |Pr[D^{F_{k}(\cdot)}(1^{n})=1]-Pr[D^{f(\cdot)}(1^{n})=1]|\le negl(n) ∣Pr[DFk​(⋅)(1n)=1]−Pr[Df(⋅)(1n)=1]∣≤negl(n)

Combining the above four equations, we finally get

Pr[Mac-forge_{\cal{A},\pi}(n)=1]\le 2^{-n}+negl(n)

which proves the theorem.

Note:

(1) Pr[Repeat]Pr[Repeat]Pr[Repeat] is negligible. CBC-MAC

Let FFF be a pseudorandom function, and fix a length function ℓ>0\ell >0ℓ>0. The basic CBC-MAC construction is as follows:

Authenticated encryption

Definitions

Let Π=(Gen,Enc,Dec)\Pi=(Gen,Enc,Dec)Π=(Gen,Enc,Dec) be a private-key encryption scheme, A\mathcal{A}A be an adversary, and nnn be the security parameter. The unforgeable encryption experiment Enc−ForgeA,π(n)Enc-Forge_{\mathcal{A},\pi}(n)Enc−ForgeA,π​(n) is defined as follows:

  1. Run Gen(1n)Gen(1^n)Gen(1n) to obtain a key kkk.
  2. The adversary A\mathcal{A}A is given input 1n1^n1n and access to an encryption oracle Enck(⋅)Enc_{k}(\cdot)Enck​(⋅). The adversary outputs a ciphertext ccc.
  3. Let m=Deck(c)m=Dec_{k}(c)m=Deck​(c), and let Q\mathcal{Q}Q denote the set of all queries that A\mathcal{A}A asked its encryption oracle. The output of the experiment is 1 if and only if (1) m≠errorm\ne \text{error}m​=error and (2) m∉Qm\notin \mathcal{Q}m∈/​Q.

A private-key encryption scheme Π\PiΠ is unforgeable if for all probabilistic polynomial-time adversaries A\mathcal{A}A, there is a negligible function neglneglnegl such that:

Pr[Enc−ForgeA,π(n)=1]≤negl(n)Pr[Enc-Forge_{\mathcal{A},\pi}(n)=1]\le negl(n) Pr[Enc−ForgeA,π​(n)=1]≤negl(n)

And a private-key encryption scheme Π\PiΠ is authenticated encryption scheme if it is CCA-secure and unforgeable.

Generic constructions

Let ΠE=(Enc,Dec)\Pi_{E}=(Enc,Dec)ΠE​=(Enc,Dec) be a CPA-secure encryption scheme and ΠM=(Mac,Vrfy)\Pi_{M}=(Mac,Vrfy)ΠM​=(Mac,Vrfy) denote a message authentication code, where key generation in both schemes simply involves choosing a uniform n-bit key. There are three natural approaches to combining encryption and message authentication using independent keys kEk_EkE​ and kMk_MkM​ for ΠE\Pi_{E}ΠE​ and ΠM\Pi_{M}ΠM​:

  1. Encrypt-and-authenticate: Given a message mmm, the transmitted value is <c,t><c,t><c,t> where

    c\leftarrow Enc_{k_E}(m)~\text{and}~t\leftarrow Mac_{k_M}(m)$$.

  2. Authenticate-then-encrypt: A MAC tag t←MACkM(m)t\leftarrow MAC_{k_M}(m)t←MACkM​​(m) is first computed, then m∣∣tm||tm∣∣t is encrypted and the resulting value EnckE(m∣∣t)Enc_{k_E}(m||t)EnckE​​(m∣∣t) is transmitted. This combination does not necessarily yield an authenticated encryption scheme, since the decryption has two steps.

  3. Encrypt-then-authenticate: The message mmm is first encrypted by c←EnckE(m)c\leftarrow Enc_{k_E}(m)c←EnckE​​(m), then the a MAC is computed by t←MackM(c)t\leftarrow Mac_{k_M}(c)t←MackM​​(c), and the ciphertext is <c,t><c,t><c,t>. If ΠE\Pi_EΠE​ is a CPA-secure private-key encryption scheme, and ΠM\Pi_{M}ΠM​ is a strongly secure MAC, then the encrypt-then-authenticate approach is an authenticated encryption scheme. Note that the kEk_EkE​ and kMk_MkM​ must be independent to ensure CCA-secure, and this is a basic principle of cryptography: different instances of cryptographic primitives should always use independent keys.

Secure communication sessions

The authenticated encryption scheme does not suffice the following potential attacks:

  • Re-ordering attack: An attacker can swap the order of the messages.
  • Replay attack: An attacker can replay a valid ciphertext ccc sent previously by one of the parties.
  • Reflection attack: An attacker can take a ciphertext ccc sent from A to B and send it back to A.

To prevent the above attacks, a counter is used to prevent the first two and a directionality bit is used to prevent the third.
